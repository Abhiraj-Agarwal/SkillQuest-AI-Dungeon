# SkillQuest: The AI Dungeon

A browser game that teaches Data Structures & Algorithms (DSA) by turning practice questions
into RPG combat. Every "monster" is a question generated by AI on the fly; answering well
deals damage, answering poorly doesn't. The game tracks per-topic accuracy and quietly steers
players toward whatever they're weakest at.

## How it plays

1. **Create a character** — username only, no password (see [Player identity](#player-identity)).
2. **Walk the dungeon map** — 11 DSA topics as connected rooms; a room unlocks once its
   prerequisite topics are solid (see [Knowledge graph](#knowledge-graph--data-model)).
3. **Fight** — entering a room generates a fresh AI question at a difficulty chosen from
   recent accuracy on that topic.
4. **Answer in free text** — an AI grader judges meaning, not exact wording: correct / partial
   / incorrect.
5. **Damage follows the verdict** — correct = double damage, partial = normal, incorrect = none.
   Clear a room's enemies to unlock the next one; clear every topic to open the boss.
6. **Guild raids** — a party shares one boss's HP pool, each member chipping away using their
   own weakest topic.
7. **Dashboards** — a stats page and an "AI Core" panel expose per-topic accuracy, the
   knowledge graph state, and submission history.

## Architecture

```text
SkillQuest/
|-- frontend/   Next.js website — no direct AI or DB access
|-- backend/    FastAPI game server — API, SQLite, game rules, default AI implementation
`-- services/   Optional standalone AI engine (more advanced grading, not used by default)
```

The frontend only ever calls the backend's REST API (`frontend/lib/api/client.js` is the one
file that calls `fetch`). The backend is the single source of truth for game state and is the
only thing that touches the database or an LLM.

**Two AI implementations exist by design.** `backend/routes/ai_real.py` calls Gemini directly
and is what runs by default — one process, no extra dependencies. `services/` is a separate,
more sophisticated implementation of the same four capabilities (real sentence-embedding
similarity instead of word-overlap fallback grading). It's off by default; point the backend
at it by running it on port 8001 and setting `AI_SERVICE_URL=http://localhost:8001` — see
[`services/README.md`](services/README.md).

## Run it locally

**Backend** (start first):

```powershell
cd backend
if (-not (Test-Path .venv\Scripts\python.exe)) { py -3.12 -m venv .venv }
& .\.venv\Scripts\Activate.ps1
python -m pip install -r requirements-dev.txt
if (-not (Test-Path .env)) { Copy-Item .env.example .env }
python -m uvicorn main:app --reload --port 8000
```

Set `GEMINI_API_KEY` in `backend/.env` (get one at [Google AI Studio](https://aistudio.google.com/apikey)).
Runs at `http://localhost:8000`, auto-creates and seeds `backend/skillquest.db` with a demo
dungeon and player on first launch. Docs at `http://localhost:8000/docs`.

**Frontend:**

```powershell
cd frontend
npm install
if (-not (Test-Path .env.local)) { Copy-Item .env.local.example .env.local }
npm run dev
```

Runs at `http://localhost:3000`. Without the backend running, every action fails with
"Could not reach the backend."

### Player identity

Username only, no password — a deliberate simplification for this build, not a real auth
system. Don't reuse a real password as your username.

## Knowledge graph & data model

```text
arrays (no prerequisites)
├── linked_lists, stacks_queues, binary_search, recursion
    ├── trees (needs linked_lists + recursion)
    │   ├── binary_search_tree, heaps, graphs (each needs trees, +binary_search for BST)
    ├── dynamic_programming (needs recursion + arrays)
    └── sorting_algorithms (needs arrays + recursion)
```

A topic unlocks once every prerequisite has `recent_accuracy > 0.65`; the boss unlocks once
every topic clears that bar. Defined in `backend/services/knowledge_graph.py`.

State lives in SQLite: `players`, `accuracy_history` (rolling last-5 accuracy per player per
topic — the number that drives unlocking, routing, and difficulty), `dungeons`/`rooms`,
`questions`, `answer_submissions`, `guilds`, `game_sessions`.

## AI logic

| System | How it works |
|---|---|
| Question generation | Gemini prompt → question + expected answer + hint, per topic/difficulty. Always fresh, never a static bank. |
| Answer grading | Gemini judges semantic correctness of free text against the expected answer → `correct`/`partial`/`incorrect`. |
| Difficulty tuning | Epsilon-greedy bandit: 90% picks by recent accuracy (>80%→hard, >50%→medium, else easy), 10% random exploration. Boss fights are always hard. |
| Topic routing | Recommends the weakest currently-unlocked topic. |

**Fallbacks** (no AI-dependent step can hard-fail): question generation retries Gemini 3× with
backoff, then falls back to a templated question. Grading falls back to word-overlap scoring.
Difficulty and topic routing never call an LLM at all. Empty answers are always scored
incorrect locally.

## API reference

Full request/response schemas are served live at `http://localhost:8000/docs`.

| Endpoint | Purpose |
|---|---|
| `POST /game/player/create` | Create a player |
| `GET /game/player/{id}` / `by-username/{username}` | Fetch stats + accuracy history |
| `GET /game/dungeons` / `dungeon/{id}` | List / fetch a dungeon and its rooms |
| `POST /game/session/start` | Start a run, bump login streak, open first room |
| `POST /game/room/enter` | Enter a room — picks difficulty, generates the question |
| `POST /game/answer/submit` | **Core loop**: grade, apply XP/damage, update accuracy, check room/dungeon completion |
| `POST /game/hint/use` | Spend a hint token |
| `GET /game/dungeon/{id}/next-topic` | Recommend next room |
| `POST /game/guild/create` / `join` | Create / join a guild |
| `GET /game/guild/{id}` | Guild + members |
| `POST /game/guild/raid/join` / `submit` · `GET /raid/status` | Raid lifecycle |
| `GET /game/leaderboard` / `leaderboard/guild` | XP rankings |
| `POST /ai/question/generate` · `/ai/answer/judge` · `/ai/difficulty/next` · `/ai/graph/next-topic` | Direct access to each AI subsystem |
| `GET /ai/dashboard/{player_id}` | Full AI Core dashboard payload |
| `GET /health`, `GET /`, `GET /docs` | Liveness, app info, Swagger UI |

- **Backend layout**: `routes/` = HTTP handlers only; `services/` = pure game/AI logic
  (no DB, no HTTP); `models/` = SQLAlchemy tables; `schemas/` = Pydantic request/response
  shapes. Game rules (XP, damage, streaks) live in `backend/services/game_logic.py` and are
  fully unit-testable without a server.
- **Frontend layout**: every backend call goes through `frontend/lib/api/client.js` — add new
  API calls there, not inline in components. `lib/statMap.js` and `lib/graphLayout.js` mirror
  backend topic/knowledge-graph constants; keep them in sync if `knowledge_graph.py` changes.
- **Env vars**: see `backend/.env.example` for what the default backend reads (`GEMINI_API_KEY`,
  `LLM_MODEL`, `DATABASE_URL`, `FRONTEND_ORIGINS`, `AI_SERVICE_URL`, `RL_*` bandit thresholds).
  `services/.env.example` has its own separate set (adds `JUDGE_*` grading thresholds) — the
  two apps don't share config.
- **Tests**:
  ```powershell
  cd backend
  & .\.venv\Scripts\Activate.ps1
  python -m pytest
  ```
  `backend/tests/test_progression.py` covers room unlocking, room/dungeon completion, and
  graph gating against an in-memory DB — no server or API key needed. `services/` has its own
  suite (see its README) covering the grader, tuner, and graph in isolation.
- **CORS**: the backend only allows credentialed requests from `FRONTEND_ORIGINS` — update it
  (comma-separated) if you deploy the frontend somewhere other than `localhost:3000`.
- Code comments referencing "Person 1/2/3" reflect original module ownership during
  development (frontend / game server / AI); they're a map of who to ask about what, not an
  access boundary.
